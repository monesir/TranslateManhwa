# خطة طبقة مترجم الذكاء الاصطناعي

## الهدف

طبقة مترجم الذكاء الاصطناعي ليست زر ترجمة إضافي. هي طبقة مسؤولة عن تحويل نتائج OCR وسياق المشروع والقاموس وترجمة Microsoft المساعدة إلى ترجمة عربية قابلة للتحرير والنشر داخل منصة ترجمة المانهوا.

الهدف التنفيذي هو بناء عقد واضح بين البرنامج وبين أي مزود ذكاء اصطناعي لاحق، بحيث نستطيع تبديل المزود أو تحسين التعليمات أو تغيير شكل السياق دون كسر صفحة الترجمة أو قاعدة البيانات.

## القرار الأساسي

نبدأ بإنشاء بيئة مترجم مستقلة داخل:

```text
agent/
```

هذا المجلد يحتوي تعليمات تشغيل المترجم، معايير الجودة، مستويات الأسلوب، وشكل المدخلات والمخرجات. لا نربط API الآن قبل تثبيت العقد.

## لماذا نبدأ بالعقد قبل الربط

ربط الذكاء الاصطناعي مباشرة قبل تعريف شكل البيانات سيؤدي إلى نتائج يصعب اختبارها:

- مخرجات نصية حرة يصعب إدخالها في قاعدة البيانات.
- اختلاف شكل الاستجابة بين مزود وآخر.
- فقدان سبب اختيار ترجمة معينة.
- صعوبة معرفة هل الخطأ من OCR أم من السياق أم من التعليمات.
- صعوبة دعم ترجمة فصل كامل مع الحفاظ على الضمائر والمصطلحات.

لذلك نعتمد مدخلات ومخرجات منظمة، ونعتبر أي مزود AI مجرد تنفيذ خلف هذا العقد.

## دور كل ملف

## AGENTS.md

ملف الدخول العام لأي agent أو مطور أو مزود ترجمة. يشرح:

- مهمة المترجم.
- الملفات التي يجب قراءتها.
- حدود المخرجات.
- أن الاستجابة يجب أن تكون JSON مطابقًا للـ schema.
- أن ترجمة Microsoft مرجع مساعد وليست النص النهائي.

## translator-system.md

تعليمات النظام الأساسية التي تصف شخصية المترجم:

- مترجم مانهوات من الإنجليزية إلى العربية.
- يحافظ على المعنى والسياق والشخصيات.
- لا يشرح داخل الترجمة.
- لا يضيف معلومات غير موجودة.
- يستخدم الفصحى المناسبة لمستوى الترجمة المختار.
- يراعي حجم الفقاعة وقابلية النص للّصق.

## arabic-style-guide.md

قواعد الأسلوب العربي المشتركة بين كل مستويات الترجمة:

- الضمائر والجنس.
- الحوار الطبيعي.
- علامات الترقيم.
- التعامل مع الأسماء والألقاب.
- التعامل مع المؤثرات الصوتية.
- عدم الترجمة الحرفية عندما تكسر العربية.

## ocr-error-correction.md

طبقة تعليمات خاصة بأخطاء OCR المعروفة. هدفها أن يفهم المترجم الآلي النص الإنجليزي المقصود قبل الترجمة، دون تعديل `sourceText` المخزن أو إخفاء الخطأ.

تغطي:

- الحروف الكبيرة والصغيرة العشوائية داخل الكلمة مثل `DeSrEvE`.
- الكلمات المتلاصقة أو المدموجة بسبب الخطوط السميكة أو المائلة.
- التباس الأحرف مثل `I/l/1` و `O/0` و `rn/m`.
- غياب الفواصل والاقتباسات وعلامات الاستفهام.
- الجمل المقسمة على عدة أسطر أو عدة فقاعات.

القواعد الأساسية:

- لا يصحح أسماء الشخصيات أو المصطلحات أو القدرات إلا بقرينة قوية أو قاموس.
- يترجم المعنى المصحح، لكنه يعيد `sourceText` كما وصله.
- إذا كان التصحيح مؤثرًا أو غير مؤكد يضيف warning مناسب مثل `ocr_corrected` أو `ocr_uncertain`.

## translation-levels.md

ملف واحد لمستويات الترجمة الخمسة بدل ملف لكل مستوى.

السبب:

- المستويات تشترك في نفس القواعد الأساسية.
- الفرق بينها هو درجة الصياغة والبلاغة والبساطة، لا نظام ترجمة مختلف.
- ملف واحد يسهل المقارنة بين المستويات ويمنع التكرار.
- يمكن لاحقًا إضافة أمثلة كثيرة لكل مستوى داخل نفس الملف.

المستويات المعتمدة:

```text
5: فصحى أدبية رفيعة
4: فصحى أدبية عالية
3: فصحى أدبية متوسطة
2: فصحى عادية
1: فصحى بسيطة جدا
```

## quality-rubric.md

معيار مراجعة الجودة. يستخدمه المترجم الآلي لاحقًا قبل إخراج النتيجة، ويمكن استخدامه أيضًا في طبقة مراجعة مستقلة.

المعايير:

- دقة المعنى.
- اتساق أسماء الشخصيات.
- اتساق المصطلحات.
- سلامة الضمائر.
- طبيعية العربية.
- ملاءمة النص لمساحة الفقاعة.
- عدم إدخال شروح.
- عدم الاعتماد الأعمى على Microsoft.

## context-contract.md

يوثق طبقات السياق التي يستقبلها المترجم:

- Global Rules.
- Project Context.
- Chapter Context.
- Page Context.
- Text Unit Context.
- Glossary Context.
- Microsoft Translation Candidate.

هذا الملف يمنع إرسال prompt عشوائي ويجعل كل معلومة لها مكان واضح.

## request.schema.json

الشكل الرسمي للطلب الذي نرسله إلى AI.

يجب أن يدعم:

- ترجمة فقاعة واحدة.
- ترجمة صفحة.
- ترجمة فصل كامل.
- مستوى الترجمة.
- القاموس.
- الشخصيات.
- سياق العمل والفصل.
- ترجمة Microsoft كمقترح.
- ترتيب الفقاعات.
- حدود الفقاعة وحجمها.

## response.schema.json

الشكل الرسمي للاستجابة.

يجب أن يعيد:

- ترجمة لكل `textUnitId`.
- مستوى الثقة.
- تحذيرات عند وجود التباس.
- المصطلحات المستخدمة.
- اقتراحات للقاموس إن ظهرت أسماء أو مصطلحات جديدة.
- ملاحظات لا تظهر داخل الفقاعة.

## أمثلة examples

الأمثلة ليست للعرض فقط. الهدف منها لاحقًا:

- اختبار prompt.
- مقارنة المزودين.
- بناء اختبارات regression.
- كشف تغيّر جودة الترجمة بعد تعديل التعليمات.

## شكل المدخلات العام

البرنامج لا يرسل نصوص OCR وحدها. يرسل كائنًا منظّمًا مثل:

```text
job:
  scope: text_unit | page | chapter
  mode: draft | revise | final
  translationLevel: 1..5

project:
  title
  arabicTitle
  synopsis
  translationNotes

chapter:
  title
  summary
  previousSummary

glossary:
  characters
  terms

units:
  textUnitId
  pageId
  pageIndex
  sequenceIndex
  sourceText
  microsoftTranslation
  speaker
  boundingBox
  neighboringText
```

## شكل المخرجات العام

المخرجات JSON فقط، مثل:

```json
{
  "version": "1.0",
  "translations": [
    {
      "textUnitId": "unit_001",
      "sourceText": "I AM NOT THE ONE WHO WILL BE KILLING YOU.",
      "arabicText": "لست أنا من سيقتلك.",
      "confidence": 0.91,
      "warnings": [],
      "usedTerms": []
    }
  ],
  "glossarySuggestions": [],
  "runWarnings": []
}
```

## مستويات الترجمة

## المستوى 5: فصحى أدبية رفيعة

يستخدم عندما نريد صياغة عربية راقية ومكثفة، مناسبة للمشاهد الجادة والدرامية. لا يعني هذا تحويل الحوار إلى إنشاء ثقيل أو لغة قديمة.

مثال اتجاه:

```text
You dare stand before me?
أتجرؤ على الوقوف بين يدي؟
```

## المستوى 4: فصحى أدبية عالية

صياغة أدبية واضحة، أقل ترفًا من المستوى الخامس وأكثر مناسبة للحوار المستمر.

```text
You dare stand before me?
أتجرؤ على الوقوف أمامي؟
```

## المستوى 3: فصحى أدبية متوسطة

الافتراضي المقترح للمانهوا. يحافظ على جمال العربية دون تضخيم.

```text
You dare stand before me?
أتجرؤ على الوقوف أمامي؟
```

## المستوى 2: فصحى عادية

واضحة ومباشرة ومناسبة لمعظم الحوارات اليومية.

```text
You dare stand before me?
كيف تجرؤ على الوقوف أمامي؟
```

## المستوى 1: فصحى بسيطة جدا

لغة سهلة جدًا ومباشرة، تصلح للنصوص السريعة أو الأعمال الخفيفة.

```text
You dare stand before me?
كيف تجرؤ أن تقف أمامي؟
```

## العلاقة مع Microsoft Translation

ترجمة Microsoft تدخل كمرشح مساعد:

- تفيد في التقاط المعنى الأولي.
- لا تعتمد كنص نهائي.
- يمكن مخالفتها إذا كانت حرفية أو ركيكة أو أخطأت في الضمائر.
- تحفظ في `translation_candidates` كمزود مستقل.
- ترجمة AI تحفظ كمرشح مستقل بمزود AI.

## نطاقات التشغيل

## text_unit

ترجمة فقاعة أو وحدة نص واحدة. مناسب للتعديل اليدوي السريع.

يحتاج:

- الوحدة الحالية.
- القاموس المطابق.
- وحدات قريبة قبلها وبعدها.
- سياق الصفحة إن وجد.

## page

ترجمة كل وحدات صفحة واحدة. مناسب عند العمل على صفحة محددة.

يحتاج:

- كل وحدات الصفحة بترتيب القراءة.
- سياق مختصر للفصل.
- القاموس المطابق.

## chapter

ترجمة الفصل كاملًا. هذا هو الوضع الأهم للجودة، لأنه يعالج:

- استمرارية الضمائر.
- تتابع الحوار.
- أسماء الشخصيات.
- نبرة المشهد.
- الجمل المقطعة بين أكثر من فقاعة.

يجب أن يعمل بنظام batch داخلي حتى لا يصبح الطلب ضخمًا جدًا على المزود.

## مبادئ التوسعة

يجب أن تكون طبقة AI قابلة للتوسع عبر:

- إضافة مزود جديد دون تغيير صفحة الترجمة.
- تغيير prompt دون تغيير قاعدة البيانات.
- إضافة حقول سياق جديدة مع الحفاظ على التوافق.
- تشغيل مراجعة جودة مستقلة لاحقًا.
- دعم ترجمة أولية ثم مراجعة AI لاحقة.

## مرحلة التنفيذ التالية بعد إنشاء الملفات

بعد تثبيت هذا العقد، المرحلة التالية تكون:

1. إنشاء `AiTranslationService` في Electron.
2. إضافة provider registry للذكاء الاصطناعي.
3. تحويل بيانات المشروع والفصل والقاموس إلى request مطابق للـ schema.
4. حفظ نتائج AI في `translation_candidates`.
5. إضافة زر AI Translate في صفحة الترجمة.
6. لاحقًا دمج AI داخل `Henry great`.

## قرار حالي

نثبت الآن البيئة والعقد فقط. لا نربط API قبل أن نحدد بدقة:

- المزود الأول.
- طريقة تخزين المفاتيح أو الإعدادات.
- سياسة batch للفصل الكامل.
- هل AI يملأ `finalTranslation` مباشرة أم يضيف candidate فقط.

القرار المبدئي: AI يضيف candidate، ولا يغير الترجمة النهائية إلا إذا طلب المستخدم ذلك أو شغّل أمرًا واضحًا مثل تطبيق ترجمة AI على الفصل.

## حالة التنفيذ الحالية

تم تنفيذ الطبقة الأولى من AI Translation فوق هذا العقد.

الملفات التنفيذية:

- `electron/application/ai-translation-service.cjs`
- `electron/application/app-api.cjs`
- `electron/ipc.cjs`
- `electron/preload.cjs`
- `src/mock/api.ts`
- `src/types/domain.ts`
- `src/types/electron.d.ts`
- `src/App.tsx`

ما تم تنفيذه:

- بناء `AiTranslationService` مستقل عن مسار Microsoft.
- قراءة ملفات `agent/` واستخدامها كتعليمات نظام للمزود.
- إضافة `ocr-error-correction.md` إلى تعليمات المزود حتى يتعامل AI مع أخطاء OCR الشائعة مثل `DeSrEvE` والكلمات المتشابكة والجمل المقطعة، مع إبقاء `sourceText` كما هو.
- بناء request مطابق للعقد من:
  - بيانات المشروع.
  - سياق العمل.
  - بيانات الفصل.
  - وحدات OCR.
  - ترجمة Microsoft عند وجودها.
  - القاموس والشخصيات ذات الصلة.
  - ترجمات سابقة قريبة داخل الفصل.
- دعم نطاقات:
  - `text_unit`
  - `page`
  - `chapter`
- دعم مستويات الترجمة الخمسة.
- دعم batch للفصل الكامل حتى لا يرسل الفصل كله في طلب واحد.
- حفظ نتائج AI في `translation_candidates` بمزود `ai`.
- عدم تعديل `final_translation` تلقائيا.
- إضافة زر `AI Translate` في صفحة الترجمة.

## مزود AI الحالي

المزود التنفيذي الأول هو:

```text
openai_compatible
```

أي مزود يدعم `/chat/completions` يمكن استخدامه عبر متغيرات البيئة:

```text
FLORIS_AI_TRANSLATION_PROVIDER=openai_compatible
FLORIS_AI_TRANSLATION_BASE_URL=https://api.openai.com/v1
FLORIS_AI_TRANSLATION_API_KEY=...
FLORIS_AI_TRANSLATION_MODEL=...
```

بدائل مدعومة:

```text
OPENAI_BASE_URL
OPENAI_API_KEY
OPENAI_MODEL
```

يمكن تعطيل `response_format` عند استخدام مزود لا يدعمه:

```text
FLORIS_AI_TRANSLATION_RESPONSE_FORMAT=off
```

## ملف مفاتيح AI المحلي

بدلا من الاعتماد على متغيرات البيئة فقط، يدعم البرنامج ملف إعدادات محلي داخل workspace:

```text
.workspace/secrets/ai-providers.json
```

هذا المسار مستبعد من git لأن `.workspace/` مستبعدة بالكامل.

الشكل المعتمد:

```json
{
  "activeProvider": "openrouter",
  "providers": {
    "openrouter": {
      "id": "openrouter",
      "type": "openai_compatible",
      "label": "OpenRouter",
      "baseUrl": "https://openrouter.ai/api/v1",
      "model": "deepseek/deepseek-v4-flash",
      "maxTokens": 8192,
      "temperature": 0.2,
      "responseFormat": false,
      "reasoning": {
        "effort": "none",
        "exclude": true
      },
      "activeKeyId": "openrouter-main",
      "headers": {
        "HTTP-Referer": "https://florismnhar.local",
        "X-Title": "FlorisMNHar"
      },
      "keys": [
        {
          "id": "openrouter-main",
          "enabled": true,
          "value": "..."
        }
      ]
    }
  }
}
```

دعم تعدد المفاتيح:

- كل مورد يملك `keys`.
- `activeKeyId` يحدد المفتاح الذي يبدأ به البرنامج.
- إذا فشل المفتاح بسبب `401` أو `403` أو `429` وكان هناك مفتاح آخر مفعل، يحاول المفتاح التالي.
- يتم حفظ `activeKeyId` في `translation_runs.settings_json` للتتبع.
- لا تحفظ قيمة المفتاح داخل قاعدة البيانات ولا داخل سجل الترجمة.
- مع OpenRouter ونماذج التفكير، يجب دعم `reasoning`. المسار الافتراضي الحالي يعطل التفكير بـ `effort: none` و`exclude: true` لأن المطلوب JSON مباشر، لا استهلاك التوكنات في reasoning وترك `content` فارغًا.
- إذا أعاد المزود JSON غير صالح بسبب طول الاستجابة أو استهلاك الإخراج، تقسم الخدمة دفعة AI تلقائيًا إلى دفعات أصغر بدل فشل الفصل الكامل.

## حدود التنفيذ الحالية

- لا يوجد اختيار مزود من صفحة الإعدادات بعد.
- لا توجد مفاتيح مخزنة داخل قاعدة البيانات. المفاتيح الآن في ملف محلي داخل `.workspace/secrets`.
- لا توجد مراجعة جودة AI مستقلة بعد.
- لا يتم تطبيق AI على `final_translation` تلقائيا.
- لا توجد streaming progress للواجهة أثناء batch الفصل الكامل.

هذه الحدود مقصودة في هذه المرحلة حتى يبقى العقد مستقرًا قبل توسيع الواجهة والإعدادات.
